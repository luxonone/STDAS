# AI Agent Startup Context SPEC

本文定义 AI Agent 在 STDAS 中获取项目上下文的规则。该规则是 SPEC，优先级高于普通设计文档中的辅助说明。

## 设计目标

本 SPEC 的核心目标是**在保证项目理解准确的前提下最小化 token 消耗**。文档读取是获取上下文的手段，而非目的。AI Agent 应根据上下文新鲜度判断是否需要读取，而非机械执行全量读取。

## 上下文新鲜度模型

AI Agent 在执行任务前，必须先判断当前上下文的新鲜度等级，再决定读取策略：

### 冷启动（Cold Start）

**触发条件（满足任一）**：
- 新 Codex 会话或新 AI Agent 首次进入 STDAS 工作区。
- 会话主题从完全不同的领域切换（例如从前端 UI 任务切换到后端 Rust 任务）。
- Context compaction 后任务范围发生了变化，且无法确认关键文档仍在上下文中。

**读取策略**：按本文"项目级必读"和"任务级必读"章节完整执行。

### 热延续（Hot Continuation）

**触发条件（同时满足）**：
- 在同一会话中继续上一个任务，且未发生 context compaction。
- 本次任务的领域和上一次请求属于同一任务类型（例如：在上一个前端修改请求后继续提出前端调整指令）。

**读取策略**：在能够明确确认相关文档已经存在于当前上下文窗口时，可以跳过重复读取。只需读取与本次具体修改直接相关、且尚未在当前上下文中的文档。

### 微调（Minor Adjustment）

**触发条件**：
- 用户对上一次输出的微调（例如："改一下这个颜色"、"把这个字段名改掉"、"加个空格"）。
- 任务范围与上一次完全一致，不涉及新的文档分区或新的 SPEC。

**读取策略**：在能够明确确认上下文新鲜、任务范围完全一致且不涉及 SPEC、架构、Git/GitHub 或跨域规则时，可以不新增读取，直接基于现有上下文执行修改。

## 上下文新鲜度判断流程

```
收到请求
  │
  ├─ 这是新会话？ ──是──→ 冷启动：完整读取
  │
  ├─ 会话中，任务领域切换了？ ──是──→ 冷启动：完整读取
  │
  ├─ 会话中，同一任务领域延续，且可确认上下文新鲜？ ──是──→ 热延续：只读新增需求文档
  │
  └─ 会话中，对上次输出的微调，且不涉及高风险文档/操作？ ──是──→ 微调：不新增读取
```

**关键原则**：当不确定上下文是否新鲜时，必须按冷启动策略或任务路由表补读关键文档。token 成本不能优先于项目理解准确性、SPEC 约束和 Git/GitHub 安全 gate。

## 适用范围

本 SPEC 适用于：

- 新 AI Agent 会话首次进入 STDAS（冷启动）。
- 同一会话中任务领域发生切换（冷启动）。
- Context compaction 后任务范围变化，且关键文档无法确认仍在上下文中（冷启动）。
- 同一会话中延续同一任务领域（热延续）。
- 同一会话中对上次输出的微调（微调）。

如果只是回答一个不改变项目状态的轻量事实问题，可以不读取全部专题文档；但一旦准备修改文件、设计方案、执行验证或操作 Git/GitHub，就必须至少完成当前新鲜度等级对应的读取。

## 核心铁律

AI Agent 不得在以下情况跳过上下文读取：

1. 冷启动时跳过项目级必读或任务级必读。
2. 热延续或微调时，声称理解了项目边界、架构、业务语义或验证要求，但实际上下文中并不包含相关文档内容。
3. Context compaction 后，未确认关键文档仍在上下文中就继续编辑 SPEC、架构或跨域规则。

AI Agent 应避免以下冗余读取：

1. 热延续时，在已经明确确认项目级必读文档仍在当前上下文窗口中后，重复读取同一批文档。
2. 微调时，在任务范围完全一致且不涉及高风险文档或操作后，重复读取无关文档。
3. 同一会话中对同一任务域的连续修改请求之间，重复读取与本次任务无关的文档。

## 项目级必读（仅冷启动时执行）

冷启动时，必须按顺序读取：

1. [SPEC 中心](README.md) — 确认 SPEC 和普通文档的分区与升级规则。
2. 本 SPEC — 确认上下文新鲜度判断和分层读取策略。
3. [docs 入口](../README.md) — 文档分区、任务路由表和当前状态。**这是任务路由的唯一入口，所有分区导航和任务必读都从这里跳转。**
4. [AI Agent 运行时规则](../architecture-design/ai-agent-runtime-rules.md) — 修改前检查、事实来源和验证 gate。
5. [STDAS 背景知识总览](../domain-knowledge/stdas-domain-primer.md) — 业务场景和领域不变量。
6. [领域模型](../architecture-design/domain-model.md) — 能力域、核心实体和数据范围。
7. [系统架构](../architecture-design/system-architecture.md) — 当前 runtime、模块边界和未来扩展约束。
8. [项目目录结构](../project-structure.md) — monorepo 目录、源码/配置/生成物边界。

任务级必读通过 [docs 入口](../README.md) 的任务路由表定位，不再在本 SPEC 中逐条列举。每个分区 README 遵循标准结构（定位 → 文档索引 → 任务必读 → 变更原则），AI Agent 读到分区 README 即可获得完整的导航信息。

这些文档提供项目背景、业务语义、系统边界和 AI 执行 gate。它们不能被 memory 或旧会话经验替代。

**热延续和微调时，只有在能够明确确认以上文档已在当前上下文窗口中时，才可以跳过重复读取。**

## 任务级必读

冷启动完成项目级必读后，任务级必读**通过 [docs 入口](../README.md) 的任务路由表定位**。不再在本 SPEC 中逐条列举。

任务路由表按任务类型（领域术语、架构边界、前端代码、Rust 后端、API 契约等）直接指向对应的分区 README 和必读文档。每个分区 README 遵循标准结构（定位 → 文档索引 → 任务必读 → 变更原则）。

如果任务同时跨多个类型，必须合并读取路径，不能只选其中一个。

**热延续时**：如果某任务类型的专题文档已在当前上下文中，可以跳过该部分；如果任务领域切换、context compaction、SPEC/架构/Git 相关内容发生变化，必须按任务路由表补读。

## 上下文变更触发重读

以下情况会**重置上下文新鲜度**，需要按冷启动策略重新读取：

1. **新会话开始**：任何新的 Codex/AI Agent 会话。
2. **任务领域切换**：从一种任务类型（如前端）切换到另一种（如后端 Rust），且切换后领域的文档不在当前上下文中。
3. **Context compaction 或 session 断裂**：无法确认关键文档仍在上下文中。
4. **SPEC 本身被修改**：如果本 SPEC 或其他 SPEC 发生了变更，必须重新读取变更后的 SPEC。
5. **跨工作日或长时间中断后**：如果会话中断超过合理时间，优先假设上下文不新鲜。

以下情况**不会**触发重读：

1. 同一会话中对同一任务领域的连续修改请求。
2. 对上一次 AI 输出的微调或延续指令，且上下文新鲜、任务范围完全一致、不涉及高风险文档或操作。
3. 简单的事实回答，不涉及文件修改。

## 读取后的执行要求

AI Agent 完成必读后（冷启动时），或确认上下文新鲜时（热延续/微调时），必须在内部判断：

1. 当前任务属于哪个 SPEC 或事实来源。
2. 是否存在用户请求和 SPEC 冲突。
3. 是否需要同步更新 docs 入口、分区 README、ADR、CHANGELOG 或交叉引用。
4. 是否需要代码验证、浏览器验证、文档引用验证或 Git 只读诊断。

如果发现冲突，必须先向用户说明冲突和推荐做法，再继续。

## memory 和旧会话的角色

- memory、rollout summary、聊天记录和旧 diff 只能帮助定位，**不得替代新会话的冷启动文档读取**。
- 但在同一会话的热延续中，如果能够明确确认上下文窗口仍包含相关文档内容，可以不重复读取。
- 如果 context compaction 后任务仍是同一任务且 summary 中明确记录了已读文档，可以根据 summary 继续，但不得假设 summary 包含了文档的完整内容。在编辑 SPEC、架构、Git/GitHub 或跨域规则前，仍必须重新读取相关关键文档。

## 例外和恢复

- 如果文档路径缺失、链接断裂或内容冲突，必须先修复文档入口或提出阻塞点，不能继续假设。
- 如果 AI Agent 无法判断上下文是否新鲜（例如不确定 compaction 是否丢失了关键内容），应按冷启动策略处理，但不需要读取已在上下文窗口中可以确认存在的文档。

## Token 效率原则

本 SPEC 的所有读取要求必须服务于实际任务需要，而非形式合规。核心原则：

1. **不重复读取已在上下文窗口中的文档**——这是最大的 token 浪费来源。
2. **冷启动时完整读取，热延续时按需增量读取，微调时只在安全条件满足时不新增读取**。
3. **读取的目的不是"完成必读清单"，而是"获得执行当前任务所需的准确上下文"**。
4. **如果 AI Agent 可以确认当前上下文中已包含所需信息，跳过读取是正确的选择，而不是违规**。

## 验收

涉及本 SPEC 的文档变更至少要执行：

```bash
rg "agent-startup-context-spec|SPEC 中心|Rust Coding Guidelines SPEC" docs CHANGELOG.md
git diff -- docs
git status --short
```

如果只是 documentation-only 变更，可以不执行代码验证，但最终回复必须说明原因。
