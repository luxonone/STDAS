# ADR-0001：STDAS 采用 Rust 原生数据平台架构

状态：Accepted
日期：2026-05-17

## 背景

STDAS 的核心复杂度来自半导体测试数据的摄入、解析、入库、分析、告警、追溯和权限治理。系统需要在保持部署简单的同时，为大规模参数数据、复杂工程分析、多客户客制化和长期演进保留空间。

## 决策

STDAS 后端采用 Rust 原生数据平台架构：

- 使用 Rust workspace 组织网关、业务服务、共享契约、基础设施库和运维工具。
- 每个业务服务都是可独立编译、独立运行、独立配置、独立健康检查的进程。
- 服务边界采用粗粒度平台能力，而不是把每个处理步骤拆成一个服务。
- PostgreSQL 作为权威业务库。
- 参数明细和大规模分析预留 Parquet、DuckDB、ClickHouse 演进路径。
- 摄入链路按可恢复流水线、事件通知和幂等处理设计。
- 所有分析查询必须具备查询预算。
- 所有业务查询必须显式携带客户范围。
- 客户差异通过 `customer-service`、`DataProfile`、规则复用/分叉和客户扩展隔离承载。

## 后果

正面：

- 领域边界清晰，适合长期演进。
- Rust 类型系统可显式表达关键业务约束。
- 网关、客户专属、数据流水线、分析、工作流和集成能力可独立扩展。
- 数据平台可从单机平滑演进到 OLAP 后端。

代价：

- 初期需要先建立领域模型和基础设施。
- Rust 对团队工程纪律要求较高。
- 查询预算、数据版本、幂等和观测字段必须从第一阶段开始实现。
- 服务契约、事件契约和配置拓扑必须保持文档化，否则分布式边界会失控。

## 验证方式

- 新代码必须落入明确 service crate、shared crate 或 libs crate。
- Gateway、Customer、DataPipeline、Analytics、Workflow、Integration 不得混成单一 service。
- 查询实现必须接收 `CustomerScope`。
- 重型分析必须支持异步 job 或明确拒绝。
- 摄入作业必须有状态机、幂等键和 correlation id。
- 新增客户差异必须先判断是配置、规则绑定、规则复用/分叉，还是隔离扩展。
